# Google Sheet Setup

The agent reads from and writes to a single Google Sheet. The Sheet is the source of truth for all transactions, budgets, and learned merchant mappings. The agent talks to it via `tools/sheets_client.py`.

This document covers:
1. **Fresh setup** — creating a new sheet from scratch
2. **Schema migration** — upgrading an existing sheet to the txn_id / telegram_message_id / MerchantMap layout

---

## Tab 1: `Transactions`

Append-only ledger. Header row at row 1, transactions below. Most recent transaction is the bottom row.

| Col | Header | Type | Notes |
|---|---|---|---|
| 1 | `Date` | YYYY-MM-DD | Transaction date (NOT logged-at date) |
| 2 | `Merchant` | Text | Raw merchant string from the bank, or user-edited |
| 3 | `Amount` | Number | Always positive |
| 4 | `Currency` | Text | Default `SGD` |
| 5 | `Category` | Text | Must match a row in the `Budget` tab. Auto-created with $0 limit if missing. `UNCATEGORIZED` is a reserved value for rows awaiting user categorisation — see "Pending categorisation" below. |
| 6 | `Source` | Text | One of: `email`, `manual`, `backfill` |
| 7 | `Payment Method` | Text | e.g. `DBS/POSB card ending XXXX`, `PayLah! Wallet (Mobile ending XXXX)`, `UOB Card ending XXXX` |
| 8 | `Notes` | Text | Optional |
| 9 | `txn_id` | Text | Canonical key. Format: `txn_<YYYYMMDD>_<NNN>`. Generated by `log_expense`. |
| 10 | `telegram_message_id` | Text | Telegram message_id of the bot's confirmation bubble. Empty for webhook-direct or backfill rows until a confirmation is sent. |
| 11 | `idempotency_key` | Text | SHA-256 hash (truncated to 16 hex chars) of `date|merchant|amount|payment_method`. Prevents duplicate inserts when the same webhook fires twice. Generated automatically by `append_transaction`. |

**Header row exactly:**
```
Date | Merchant | Amount | Currency | Category | Source | Payment Method | Notes | txn_id | telegram_message_id | idempotency_key
```

The header strings must match exactly (case-sensitive) — `tools/sheets_client.py` looks up columns by header name via `_get_column_index`.

## Tab 2: `Budget`

The agent supports two layouts. Pick one and stick with it.

### Per-month layout (recommended)

Row 1 is the header, columns 2–13 are the months.

| Col 1 | Col 2 | Col 3 | … | Col 13 |
|---|---|---|---|---|
| `Category` | `Jan` | `Feb` | … | `Dec` |

The agent's `read_budgets(month_num)` and `update_budget_row(category, limit, month_num)` write to the appropriate month column. When the user says "set Food budget to $500 for June," the June column gets updated.

### Simple layout

Two columns: `Category | Monthly Limit`. The same limit applies all year.

The agent auto-detects the layout by counting header columns: 13+ columns = per-month, otherwise simple.

### Default categories

A starting set is in `budget_categories.json` — `Food & Dining`, `Transport`, `Groceries`, `Shopping`, `Entertainment`, `Subscriptions`, `Utilities`, `Health`, `Miscellaneous`. Treat it as a generic seed for new installs; the agent doesn't depend on these specific names — it picks whatever rows exist in the `Budget` tab and auto-creates new ones at $0 limit when a transaction lands in an unseen category.

## Tab 3: `MerchantMap` (new in v4)

Stores learned merchant → category mappings so the agent doesn't have to re-ask after corrections. Created automatically by `add_merchant_mapping` on first write — you don't have to make it manually, but if you want to seed it, here's the layout:

| Col | Header | Type | Notes |
|---|---|---|---|
| 1 | `merchant_pattern` | Text | Substring matched case-insensitively against incoming merchant strings |
| 2 | `category` | Text | Must match a category in the `Budget` tab |
| 3 | `created_at` | YYYY-MM-DD | When the mapping was learned |

**Header row exactly:**
```
merchant_pattern | category | created_at
```

Lookup rule: longest matching substring wins. So `GRABFOOD` beats `GRAB` if both match. Be specific enough to avoid collisions but generic enough to generalise (e.g. `SHOPEE SINGAPORE` rather than the full `SHOPEE SINGAPORE MP`).

The `expense-tracker` skill calls `lookup_merchant_category` BEFORE asking the user to disambiguate, and calls `learn_merchant_mapping` AFTER the user makes a correction.

## Pending categorisation (`UNCATEGORIZED`)

When a transaction arrives but categorisation is ambiguous (always-ask merchants, PayLah!/PayNow transfers to individuals, low-confidence matches), the agent calls `log_expense_pending` instead of `log_expense`. This logs the row immediately with `Category = UNCATEGORIZED` and sends a Telegram ask-prompt with numbered options. The row exists in the ledger so the transaction is never lost even if the user never replies.

**Effects of `UNCATEGORIZED`:**

- Excluded from `get_spending_summary`, `generate_spending_report`, and `detect_subscription_creep` aggregates — they don't count toward category totals until the user picks a real category.
- Surfaced in `generate_spending_report.pending_review` as a list of rows awaiting review, so the LLM can nag about them in summaries.
- When the user replies to the ask-prompt, the skill resolves the `txn_id` via reply-to-message and calls `edit_expense(txn_id=..., new_category=...)` — the row stays in the ledger, just with a real category.

No manual action required. The `UNCATEGORIZED` row in the `Budget` tab is auto-created with $0 limit on first use of `log_expense_pending`.

## Tab 4: `Insights` (new in v4.5)

Tier-4 semantic memory — derived facts from spending reports persisted for future reference. Auto-created on first `write_insight` call.

| Col | Header | Type | Notes |
|---|---|---|---|
| 1 | `date` | YYYY-MM-DD | When the insight was written |
| 2 | `category` | Text | One of: `spending`, `budget`, `subscription`, `trend`, `general` |
| 3 | `month` | YYYY-MM | Month the insight relates to |
| 4 | `insight` | Text | Short factual statement |

**Header row exactly:**
```
date | category | month | insight
```

The agent writes 2–3 insights after each spending report. Before generating a new report, it reads prior insights via `get_insights` so summaries build on prior conclusions.

## Tab 5: `Journal` (new in v4.8 — reply = journal experiment)

Free-text narrative replies from the user to bot-initiated summary messages (9 PM daily review, weekly summary, monthly report) land here, keyed by date. Auto-created on the first `append_journal_entry` call — you don't need to make it manually.

| Col | Header | Type | Notes |
|---|---|---|---|
| 1 | `date` | YYYY-MM-DD | Date the reply was written (not the date any referenced transaction occurred) |
| 2 | `reply_text` | Text | User's reply verbatim |
| 3 | `txn_ids_referenced` | Text | Comma-separated `txn_ids` mentioned in the reply. Empty if none. |
| 4 | `tags` | Text | Optional 1-3 short tags extracted from the reply (e.g. `partner,regret,promo`). Empty is fine. |

**Header row exactly:**
```
date | reply_text | txn_ids_referenced | tags
```

The Journal tab is a 2-week experiment (as of v4.8) to test whether narrative capture via reply-to-cron-summary sticks as a habit. If it does, a later milestone will add a nightly Obsidian mirror that writes one markdown file per date into the user's vault. Until then, the sheet is the sole store.

## Tab 6: `WebhookLog` (new in v4.9 — missed-transaction sweep)

Auto-created by the Gmail Apps Script (`apps-script/Code.gs::logToAuditSheet`)
on the first parsed bank email after the Apps Script is redeployed with the
audit-log code. You don't need to create it manually.

The Apps Script writes every parsed bank email here BEFORE firing the webhook,
so the parsed payload is durable even if the webhook / LLM call fails. The
`sweep_missed_transactions` tool later diffs this tab against `Transactions`
by `idempotency_key` and surfaces anything that was dropped.

| Col | Header | Type | Notes |
|---|---|---|---|
| 1 | `timestamp` | ISO 8601 | When the Apps Script processed the email |
| 2 | `bank` | Text | `DBS` or `UOB` |
| 3 | `type` | Text | `paylah` or `card` |
| 4 | `amount` | Number | Parsed transaction amount |
| 5 | `currency` | Text | `SGD` |
| 6 | `merchant` | Text | Raw merchant string from the bank email |
| 7 | `date` | YYYY-MM-DD | Transaction date |
| 8 | `payment_method` | Text | Card / wallet identifier |
| 9 | `idempotency_key` | Text | Same 16-char SHA-256 truncation as `Transactions.idempotency_key` — this is the join key for the sweep |
| 10 | `webhook_status` | Text | `pending` → `sent` / `failed:<code>` / `error:<msg>` after the webhook call returns |
| 11 | `matched` | Text | `yes` once a prior sweep has reconciled this row against a Transactions entry; empty otherwise |

**Header row exactly:**
```
timestamp | bank | type | amount | currency | merchant | date | payment_method | idempotency_key | webhook_status | matched
```

**Apps Script setup requirements:**

- `SPREADSHEET_ID` must be set in Apps Script → Project Settings → Script
  Properties (same Google Sheet ID you used for `GSPREAD_SPREADSHEET_ID` in
  `~/.hermes/.env`).
- The Apps Script runs under your Google account, which already has Editor
  access to the sheet, so no extra sharing is needed.

**Idempotency key parity:** the JS `computeIdempotencyKey` in `Code.gs` must
produce the same 16-char hex as Python's
`tools.sheets_client._compute_idempotency_key` for the same input. Run
`testIdempotencyKeyParity()` once from the Apps Script editor after
redeploying — it asserts the locked-in values also pinned in
`tests/test_expense_sheets_tool.py::TestIdempotencyKeyParity`.

## Tab 7: `Cards` (new in v5 — card optimiser)

Per-card metadata for the card optimiser. The `card-optimiser` skill reads
this tab to resolve `Payment Method` strings back to a `card_id` for cap
tracking and nudges.

**You must populate this manually.** The card optimiser stays inert
(`status: "setup_required"`) until at least one row exists here and one in
`CardStrategy`.

| Col | Header | Type | Notes |
|---|---|---|---|
| 1 | `card_id` | Text | Stable lowercase-hyphenated key, e.g. `dbs-altitude`, `uob-prvi`, `citi-pm`. Referenced by `CardStrategy.primary_card_id` / `fallback_card_id`. |
| 2 | `display_name` | Text | Human-readable name shown in nudges, e.g. `DBS Altitude Visa`. |
| 3 | `payment_method_pattern` | Text | Case-insensitive substring matched against `Transactions.Payment Method`. **Longest match wins** when multiple patterns match. Must match what the bank emails actually produce (e.g. `DBS/POSB card ending XXXX`). |
| 4 | `cycle_start_day` | Integer 1–31 | Day the statement cycle resets. Clamped to `min(day, days_in_month)` for short months (Feb, Apr, etc.). |
| 5 | `min_spend_bonus` | Number | Optional minimum spend to unlock the headline rate. `0` or blank = no minimum. |
| 6 | `notes` | Text | Free-text quirks (e.g. "floor not ceiling", "promo until Jun"). |

**Header row exactly:**
```
card_id | display_name | payment_method_pattern | cycle_start_day | min_spend_bonus | notes
```

## Tab 8: `CardStrategy` (new in v5 — card optimiser)

Which card to use for which category, plus cap / earn-rate / fallback.
The card optimiser looks up strategy by exact category match first, then
falls back to the `_default` sentinel row.

**You must populate this manually, including a `_default` row.** Without
the sentinel, unmapped-category transactions bubble a "no strategy" error.

| Col | Header | Type | Notes |
|---|---|---|---|
| 1 | `category` | Text | Must match a `Budget` tab category exactly, or the sentinel `_default`. |
| 2 | `primary_card_id` | Text | Key from the `Cards` tab. |
| 3 | `primary_cap` | Number | Monthly-cycle cap at headline rate. `0` or blank = unlimited. |
| 4 | `primary_earn_rate` | Number | Miles per dollar (mpd) at headline rate. |
| 5 | `fallback_card_id` | Text | Key from the `Cards` tab. Optional. |
| 6 | `fallback_earn_rate` | Number | mpd on fallback. Optional. |
| 7 | `promo_active_until` | YYYY-MM-DD | Only set via `set_category_primary` override; blank for default rows. The lazy reversion in `plan_month` restores the prior config after this date. |
| 8 | `notes` | Text | Free text. Also stores `||PREV:<snapshot>` when a promo override is active — used for reversion. Don't edit that prefix manually. |

**Header row exactly:**
```
category | primary_card_id | primary_cap | primary_earn_rate | fallback_card_id | fallback_earn_rate | promo_active_until | notes
```

**Sentinel row required.** Exactly one row with `category = "_default"` for
unmapped-category spend. Example:

| category | primary_card_id | primary_cap | primary_earn_rate | fallback_card_id | fallback_earn_rate | promo_active_until | notes |
|---|---|---|---|---|---|---|---|
| `_default` | `uob-prvi` | 0 | 1.4 | | 0 | | general-spend catch-all |

## Tab 9: `CardNudgeLog` (auto-created)

Auto-created by `card_optimiser._ensure_nudge_log` on the first nudge.
You don't need to create it manually. Records which nudges have already
fired to prevent re-firing within the same cycle.

| Col | Header | Type | Notes |
|---|---|---|---|
| 1 | `timestamp` | ISO 8601 | When the nudge was sent. |
| 2 | `cycle_window` | Text | `YYYY-MM-DD_YYYY-MM-DD` — start/end of the cycle window the nudge fired in. |
| 3 | `card_id` | Text | The card whose cap was hit. |
| 4 | `category` | Text | The category whose cap was hit. |
| 5 | `threshold` | Integer | `80` or `100` — which threshold crossing fired the nudge. |
| 6 | `triggering_txn_id` | Text | Best-effort txn_id of the transaction that pushed over the threshold. |
| 7 | `fallback_card_id` | Text | The card the nudge suggested switching to. |

**Header row exactly:**
```
timestamp | cycle_window | card_id | category | threshold | triggering_txn_id | fallback_card_id
```

Dedup rule: at most one nudge per `(cycle_window, card_id, category, threshold)` tuple. The 100% nudge is additionally suppressed if the 80% nudge already fired AND the last 2 transactions in that category used the fallback card ("silent when already switched").

## Sharing with the service account

After creating the sheet:

1. Copy the spreadsheet ID from the URL: `https://docs.google.com/spreadsheets/d/<THIS_IS_THE_ID>/edit`
2. Click **Share** and add the service account email (from your service-account JSON, looks like `name@project.iam.gserviceaccount.com`)
3. Give it **Editor** access
4. Put the spreadsheet ID in `~/.hermes/.env` as `GSPREAD_SPREADSHEET_ID`

## Schema migration: existing sheet → v4 schema

If your sheet was created before the txn_id / telegram_message_id / MerchantMap columns existed, follow these steps to upgrade. The code in `tools/sheets_client.py` is forward-compatible — it tolerates missing columns gracefully — but `edit_expense` and `delete_expense` only use `txn_id` lookup once the column is present.

### Step 1 — Add the new columns to `Transactions`

1. Open the `Transactions` tab.
2. In row 1, add three new headers in columns 9, 10, and 11 (immediately after `Notes`):
   - Column 9 header: `txn_id`
   - Column 10 header: `telegram_message_id`
   - Column 11 header: `idempotency_key`
3. Leave the cells in those columns empty for existing rows for now — Step 2 fills them in.

### Step 2 — Backfill `txn_id` for existing rows

For each existing transaction row, fill the `txn_id` cell with a value of the form `txn_legacy_NNN` (3-digit sequence in insertion order). Example for the 5 existing rows:

| Row | txn_id |
|---|---|
| 2 | `txn_legacy_001` |
| 3 | `txn_legacy_002` |
| 4 | `txn_legacy_003` |
| 5 | `txn_legacy_004` |
| 6 | `txn_legacy_005` |

Why `txn_legacy_NNN` instead of generated `txn_<YYYYMMDD>_<NNN>` IDs? It makes legacy rows visually distinct from rows generated post-migration, which helps when debugging. The format is your call — anything unique works, since the column is just text.

Leave `telegram_message_id` and `idempotency_key` empty for legacy rows — they pre-date these features. New transactions will populate them automatically.

### Step 3 — Create the `MerchantMap` tab (optional)

You can either:
- **Do nothing.** The first call to `learn_merchant_mapping` creates the tab automatically.
- **Or seed it manually.** Add a tab named exactly `MerchantMap` with the header row `merchant_pattern | category | created_at`, then add rows for any mappings you already know you want (e.g. `GRABFOOD → Personal - Food & Drinks`).

### Step 4 — Verify

After migration, the agent should:
- Generate `txn_<YYYYMMDD>_<NNN>` IDs on new transactions.
- Successfully edit a legacy row when given its `txn_legacy_NNN` ID.
- Fall back to merchant+amount lookup if asked to edit a row with no `txn_id`.
- Return a `MerchantMap` hit on the second-ever call to `lookup_merchant_category` for a merchant you've corrected.
